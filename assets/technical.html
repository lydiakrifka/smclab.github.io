0. Dev Environment: uv & pixi

Hardware context: daily drivers are a System76 Oryx Pro, an Alienware Area-51 with a 5080 GPU, plus a PinePhone Pro and a Google Pixel running GrapheneOS. I’ve lived through pipenv, pyenv, poetry, conda/mamba, and a small zoo of virtualenv hacks. This setup currently wastes the least time.

Latest pro tip: use uv or pixi

I’ve tried most of the usual Python environment managers. My current conclusion is straightforward: use uv for almost everything, keep pixi in reserve for ML stacks that really need Conda-style system libraries.

The three recurring pain points:

  1. Managing project environments and Python versions.
  2. Installing isolated CLI tools.
  3. AI/ML setups tied to CUDA / ROCm / BLAS / other system libs.

uv covers (1) and (2) cleanly and handles (3) surprisingly well for modern projects. When it doesn’t, that’s the moment to reach for pixi, not resurrect a global Conda base.

1. Development with Python

For normal projects (DSP notebooks, MIR scripts, small tools) I just use uv:

  • uv init — create a new project with a pyproject.toml.
  • uv pip — drop-in replacement for pip in legacy projects.
  • uv python — install and select Python versions per user, without stacking another manager on top.

This replaces the old chain of pipenv, pyenv, poetry, random virtualenvs, and half-remembered shell functions. One tool, one mental model.

2. Installing CLI tools written in Python

For Python-based CLI tools I no longer bother with pipx. I let uv do it:

  • uv tool install <package> — install into an isolated env.
  • Each tool gets its own virtual environment and can have a pinned Python version if needed.

Result: less global Python clutter, fewer “which pip installed this?” archaeology sessions.

3. AI / ML with Python

For AI / ML work I start with uv and only escalate if the project actually breaks. Most newer codebases target recent Python + recent system libraries. In that world, a plain uv environment is enough.

Classic Conda/mamba argument:

  • You can pin different BLAS, CUDA, etc. versions inside the env, independent of the host system.

That’s still useful for older projects or anything that depends on a specific ROCm/CUDA/Triton combo. The average ML repository with a huge requirements.txt is often technically unsatisfiable as-is. In those cases I either patch the deps or move that project into pixi and freeze the environment there.

pixi behaves like a saner Conda: project-local envs, no global base trying to own the machine, and declarative lockfiles instead of incantations.

Rule of thumb for future Lydia:
Start with uv. If it installs cleanly and tests pass, stop thinking about it. If you’re wrestling CUDA / ROCm / Triton / ancient dependencies for more than a short while, switch that project to pixi and keep everything else boring.

2. Minimal uv Workflow

This is the minimal flow I want to remember. Everything else can be reconstructed from here. 

# 0. Make sure uv is installed (system-wide or via your package manager).
#    On Arch you probably know what you’re doing.

# 1. New project
uv init smc-experiment
cd smc-experiment

# 2. Add dependencies
uv add numpy scipy librosa scikit-learn

# 3. Pin Python version if needed
uv python install 3.11
uv python pin 3.11

# 4. Run your stuff
uv run python scripts/whatever.py

# 5. For a CLI tool (ruff, black, etc.)
uv tool install ruff
ruff check .
If a script can’t be re-run on a different day, on a different machine, without fighting the environment, it doesn’t belong in this notebook yet.
