0. Dev Environment: uv & pixi

Hardware context: daily drivers are a System76 Oryx Pro, an Alienware Area-51 with a 5080 GPU, plus a PinePhone Pro and a Google Pixel running GrapheneOS. I’ve lived through pipenv, pyenv, poetry, conda/mamba, and a small zoo of virtualenv hacks. This is the setup that currently wastes the least time.

Latest pro tip: use uv or pixi

I’ve tried most of the usual Python environment managers. My current conclusion is simple: use uv for almost everything, keep pixi in reserve for weird ML stacks that really need Conda-style system libraries.

Most people run into three kinds of Python packaging problems:

  1. Managing project environments and Python versions.
  2. Installing isolated CLI tools.
  3. AI/ML setups that depend on CUDA / BLAS / other system libraries.

uv solves almost all of this for day-to-day work. When it doesn’t, the fallback is pixi, not going back to a global Conda base environment.

1. Development with Python

For normal projects (DSP notebooks, MIR scripts, small web tools) I just use uv:

  • uv init — create a new project with a pyproject.toml.
  • uv pip — drop-in replacement for pip in legacy projects.
  • uv python — manage installed Python versions, per user, without extra pyenv layers.

This replaces the old stack of pipenv, pyenv, poetry, random virtualenvs, and half-remembered shell functions. One tool, one mental model.

2. Installing CLI tools written in Python

For Python-based CLI tools, I no longer use pipx. I let uv handle that too:

  • uv tool install <package> — install into an isolated environment.
  • Each tool gets its own virtual environment and pinned Python version if needed.

Result: less global Python mess, no “which pip installed this thing” archaeology.

3. AI / ML with Python

For AI / ML work I start with uv and only escalate if needed. Most newer projects target recent Python versions and up-to-date system libraries. In those cases, a normal uv environment is enough.

The classic argument for Conda/mamba is:

  • You can pin different versions of BLAS, CUDA, etc. inside the env, independently of the system.

That’s still valid for older or finicky ML projects, or anything that needs specific ROCm / CUDA combos or obscure binary dependencies. For those: use pixi.

pixi behaves like a saner Conda: project-local envs, no global Conda base trying to manage your universe, and declarative lockfiles instead of manual spell-casting.

Rule of thumb for future Lydia:
Start with uv. If the project installs cleanly and runs tests, stop there. If you’re fighting CUDA / ROCm / Triton / ancient dependencies for more than a few minutes, switch that project to pixi and keep the rest of the system simple.

2. Minimal uv Workflow

The only “snippet” I really need here is a reminder of the minimal flow for spinning up a new experiment. Everything else can be reconstructed from this. 

# 0. Make sure uv is installed (system-wide or via your package manager).
#    On Arch you probably know what you’re doing.

# 1. New project
uv init smc-experiment
cd smc-experiment

# 2. Add dependencies
uv add numpy scipy librosa scikit-learn

# 3. Pin Python version if needed
uv python install 3.11
uv python pin 3.11

# 4. Run your stuff
uv run python scripts/whatever.py

# 5. For a CLI tool (ruff, black, etc.)
uv tool install ruff
ruff check .
